ops-hub의 데이터 페칭 표준 원칙(2026-06-26 사용자 확정): **백엔드 응답은 화면 렌더에 꼭 필요한 최소 정보만** 담고, **필터링은 서버에서** 수행한다. 클라이언트 편의(즉시 토글)를 위해 부가 속성(예: 타인의 `jobFunction` 등 HR 속성)을 페이로드에 실어 내려보내지 않는다.

연차 캘린더 직무 필터에서 처음 합의됨 — 클라이언트 토글(전체 이벤트 + jobFunction 동봉 후 브라우저 필터) 대신 **서버가 직무로 거른 결과만 반환**하고 `jobFunction`은 응답에서 제외. 적대검증이 "코워커 jobFunction 추가 노출"을 지적했고, 사용자는 차후 **업무·캘린더·대시보드까지 동일 규칙**으로 가길 원함.

**Why:** 데이터 최소화(least exposure) — 클라이언트는 신뢰 경계 밖이라 내려간 데이터는 마스킹돼도 사실상 노출. 부가 속성을 안 내려보내면 권한·프라이버시 사고 표면이 줄고, 필터 정책이 서버 한 곳에 모인다.

**How to apply:** 새 조회 API/화면 설계 시 (1) 응답 필드를 "이 화면이 실제로 그리는 데 필요한가"로 깎고, (2) 필터/검색은 서버 쿼리 파라미터로 받아 서버에서 적용하며(클릭 시 재요청 허용), (3) 권한별 마스킹뿐 아니라 "애초에 안 보내기"를 우선 고려한다. 트레이드오프(필터마다 서버 왕복)는 수용 — 캘린더는 이미 월 이동 시 재요청하므로 query key에 필터를 더하는 수준.
